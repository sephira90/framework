# 2026-03-13 Project Rules And Validation

## Context
- Уточнена стратегическая цель проекта: совместная разработка PHP-фреймворка должна одновременно развивать системное мышление, критический анализ и инженерную дисциплину.
- Потребовалось превратить эту цель из общей декларации в операционные правила проекта.
- Дополнительно потребовалось зафиксировать обязательный протокол ведения журнала решений и обязательную цепочку валидации для каждого содержательного действия.

## Key Findings
- В `AGENTS.md` уже был общий `Execution Log Protocol`, но в нём не хватало жёсткой привязки к процессу разработки фреймворка и к обязательным проверкам качества.
- В рабочем каталоге проекта пока отсутствуют `composer.json` и `vendor/`, поэтому проект ещё не оснащён зависимостями и конфигурацией инструментов статического анализа.
- Команды `php`, `phpcs`, `phpstan` и `psalm` не доступны из текущего `PATH`.
- При этом `C:\OSPanel\modules\PHP-8.4\php.exe` существует, значит проблема не в полном отсутствии PHP-рантайма, а в том, что валидационный контур ещё не собран и не подведён к проекту.

## Decisions
- В `AGENTS.md` добавлены явные системные цели фреймворка: целостность архитектуры, минимальность ядра, явные контракты, наблюдаемость, статическая проверяемость и обучающая прозрачность модулей.
- В критерии успеха добавлено требование обязательной автоматической калибровки через `php -l` / lint, `phpcs`, `phpstan` уровня `10` и `psalm`.
- `Execution Log Protocol` расширен секциями `Changed Files`, `Validation` и `Open Risks`, чтобы записи стали пригодны для инженерной реконструкции решений.
- Добавлен отдельный `Протокол разработки и валидации`, который запрещает считать изменение завершённым без явной проверки либо без документированного объяснения, почему проверка пока невозможна.

## What This Changes
- Проект теперь формально ориентирован не просто на создание PHP-фреймворка, а на создание проверяемой системы с явными архитектурными критериями.
- Любая будущая содержательная правка должна сопровождаться записью в `artifacts/execution/`.
- Отсутствие настроенных инструментов проверки теперь считается не фоновым неудобством, а отдельным блокером качества, который нужно устранить до серьёзной разработки ядра.

## Validation
- `php -l`: `N/A` для текущего изменения, потому что PHP-файлы не изменялись.
- `phpcs`: не выполнен; инструмент не доступен в текущем проекте и в текущем `PATH`.
- `phpstan --level=10`: не выполнен; инструмент не доступен в текущем проекте и в текущем `PATH`.
- `psalm`: не выполнен; инструмент не доступен в текущем проекте и в текущем `PATH`.
- Дополнительная калибровка текущего состояния среды:
  - `php -v` не запускается из `PATH`;
  - `phpcs --version` не запускается из `PATH`;
  - `phpstan --version` не запускается из `PATH`;
  - `psalm --version` не запускается из `PATH`;
  - `C:\OSPanel\modules\PHP-8.4\php.exe` существует;
  - `composer.json` отсутствует;
  - `vendor/` отсутствует.

## Open Risks
- Без `composer.json`, зависимостей и конфигурации статических анализаторов требование обязательной валидации пока нельзя исполнять автоматически на уровне проекта.
- Если начать писать ядро до сборки инструментального контура, проект быстро накопит не калиброванный архитектурный и типовой долг.

## Next Actions
- Создать минимальный `composer.json` и определить dev-зависимости для `phpcs`, `phpstan` и `psalm`.
- Зафиксировать единый способ запуска локального PHP через `C:\OSPanel\modules\PHP-8.4\php.exe`.
- Подготовить конфигурацию lint / coding standard / `phpstan` уровня `10` / `psalm`.
- Только после этого переходить к первой партии PHP-кода ядра фреймворка.
